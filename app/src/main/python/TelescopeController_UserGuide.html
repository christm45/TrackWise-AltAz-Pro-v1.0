📦 1. Installation & First Launch

Installing the APK

  1. Copy app-debug.apk to your Android device
  2. Open the file on your device. You may need to enable "Install from unknown sources" in Settings > Security
  3. Tap "Install" and wait for completion
  4. Launch "TrackWise AltAz Pro" from your app drawer

First Launch Checklist

  1. Grant Location Permission - The app needs GPS for accurate celestial coordinate calculations. Accept the permission prompt.
  2. Grant Notification Permission (Android 13+) - Needed for the foreground service notification that keeps the telescope connection alive.
  3. Set Observer Location - Go to the Location tab and either use GPS auto-detect or manually enter your latitude/longitude. Accurate coordinates are critical for Alt/Az ↔ RA/Dec conversions.
  4. Connect to Telescope - Use the Connection card on the Control tab to connect via WiFi or USB.

📡 2. Connecting to Your Telescope

The app supports two major telescope mount protocol families and two physical connection methods, giving you flexibility to work with a wide range of mounts. This chapter explains exactly how the communication works, from the WiFi radio waves down to the individual protocol commands.

How the App Communicates with Your Mount

Understanding the communication stack helps you troubleshoot connection issues and configure settings correctly. Here is the full path from the app's UI to your mount hardware:

OnStep Mounts (WiFi Connection)

OnStep is a popular open-source telescope controller running on Arduino, Teensy, or ESP32 microcontrollers. It is the primary target for this app, and all features are fully tested with OnStep firmware.

How OnStep WiFi Works

OnStep mounts typically include an ESP32 or ESP8266 (WeMos D1 Mini) WiFi module that creates a TCP server. The module can operate in two modes:

ModeHow It WorksDefault IPWhen to Use
Access Point (AP) The mount creates its own WiFi network (e.g., "OnStepX"). Your phone connects directly to it. 192.168.0.1 In the field, away from home WiFi. Most common setup.
Station (STA) The mount joins your existing home/observatory WiFi network. It gets an IP from your router. Assigned by your router (check DHCP list) In an observatory with permanent WiFi.

Step-by-Step: Connecting to OnStep

  1. Power on your mount — Wait 5-10 seconds for the WiFi module to boot. The OnStep ESP32 starts a TCP server on port 9999 (configurable in OnStep's Config.h).
  2. Connect your phone to the mount's WiFi:
    • Go to Android Settings → WiFi
    • Find the mount's network (typically named OnStepX, OnStep, or a custom name)
    • Connect to it. You will lose internet access (the mount has no internet)
  3. Open the app and go to the Connection card on the Control tab
  4. Set these values:
    • Mode: WiFi
    • Protocol: LX200 (this is the default)
    • IP Address: 192.168.0.1 (or your OnStep's configured IP)
    • Port: 9999
  5. Tap Connect

What Happens During Connection

When you tap Connect, the app performs this sequence (all within 5-10 seconds):

  1. TCP socket opens — The app creates a TCP connection to the IP:port you specified, with a 5-second timeout. TCP_NODELAY is enabled for low-latency command exchange.
  2. 500ms stabilization wait — Some OnStep firmware versions need a brief pause after accepting a TCP connection before they can process commands.
  3. Identification commands — The app sends:
    • :GVP# — "Get Version Product name" (OnStep returns e.g., On-Step#)
    • :GVN# — "Get Version Number" (returns e.g., 4.24q#)
  4. OnStep auto-detection — If the product name contains "OnStep" or "On-Step", the app enables OnStep-specific features (native Alt/Az GoTo, extended tracking commands, PEC, etc.)
  5. Position test — The app reads the current position via :GR# (RA) and :GD# (Dec) to verify two-way communication is working.
  6. Background threads start — Two threads begin:
    • Position polling — Reads Alt, Az, RA, Dec, and slew status every 0.5-1 second
    • Command queue — Processes your slew/goto/sync commands in order
If the mount doesn't respond to :GVP# but the TCP socket stays open, the app still accepts the connection (some OnStep configurations are "silent" until the first real command). You'll see "WiFi Telescope (OnStep)" as the mount name in this case.

OnStep-Specific Features

When OnStep is detected, these extra capabilities are unlocked:

  • Native Alt/Az GoTo — Uses :Sa# (set altitude) + :Sz# (set azimuth) + :MA# (slew horizontal). Standard LX200 only supports RA/Dec GoTo.
  • Custom tracking rates — :SXTR,{rate}# and :SXTD,{rate}# set precise RA/Dec correction rates in arcseconds/second.
  • Weather data upload — Sends temperature, pressure, and humidity to OnStep for atmospheric refraction correction (:SX9A#, :SX9B#, :SX9C#).
  • PEC (Periodic Error Correction) — Record, play, and save periodic error tables (:$QZ+#, :$QZ-#, :$QZ/#, :$QZ!#).
  • Dual focuser control — :F+#/:F-# (move), :FQ# (stop), :FG# (get position).
  • Park/Home — :hP# (park), :hF# (home).

SynScan / Sky-Watcher Mounts (WiFi Connection)

Sky-Watcher mounts with a SynScan WiFi adapter or built-in WiFi (like the AZ-GTi, Virtuoso, or Star Adventurer GTi) use the NexStar hand controller protocol. This is because Sky-Watcher (Synta Technology) is the OEM manufacturer for Celestron, so they share the same communication protocol.

How SynScan WiFi Works

The SynScan WiFi adapter (or built-in WiFi module) creates a WiFi access point. Your phone connects to it, then communicates via TCP. The protocol is different from LX200 — it uses short 1-2 character commands with hexadecimal-encoded positions instead of ASCII text.

Mount TypeWiFi ModuleDefault SSIDDefault IPTCP Port
AZ-GTi Built-in SynScan_XXXX 192.168.4.1 11882
Virtuoso / Dob GoTo Built-in SynScan_XXXX 192.168.4.1 11882
EQ6-R / HEQ5 + SynScan WiFi External dongle SynScan_XXXX 192.168.4.1 11882
Star Adventurer GTi Built-in SynScan_XXXX 192.168.4.1 11882

Step-by-Step: Connecting to SynScan

  1. Power on your mount and wait for the SynScan WiFi to start (the WiFi LED blinks, then goes steady after ~10 seconds).
  2. Connect your phone to the SynScan WiFi:
    • Go to Android Settings → WiFi
    • Find SynScan_XXXX (the last 4 digits vary per device)
    • Connect (no password by default on most models)
  3. Important: Close the SynScan Pro app if it's running. Only one app can connect to the mount at a time via TCP.
  4. Open TrackWise AltAz Pro and go to the Connection card
  5. Set these values:
    • Mode: WiFi
    • Protocol: NexStar
    • IP Address: 192.168.4.1
    • Port: 11882
  6. Tap Connect

What Happens During SynScan Connection

  1. TCP socket opens to 192.168.4.1:11882
  2. Echo test — Sends K\x55 (echo command). The mount should return U# (ASCII 'U' + '#' terminator). This confirms basic communication.
  3. Version query — Sends V. Returns firmware version. SynScan v3.38+ returns 6 ASCII hex digits (e.g., 042507# = version 4.37.7).
  4. Model identification — Sends m. Returns a model byte that identifies the mount type (AZ-GTi = 165, EQ6 = 0, HEQ5 = 1, etc.).
  5. Position test — Reads Alt/Az via z command and RA/Dec via e command (both return 32-bit hex-encoded angles).
  6. Background threads start — Same as OnStep: position polling + command queue.

SynScan-Specific Features

  • High-precision position encoding — Positions are encoded as 32-bit hex fractions of a full circle (e.g., 1234ABCD = 1234ABCD / 2^32 * 360 degrees). This gives sub-arcsecond precision.
  • Variable-rate Alt/Az tracking — The app sends per-axis correction rates in arcsec/second using the motor passthrough command. Rate = arcsec_per_sec * 4 encoded as 2 bytes. This is used by the real-time tracking controller for precise corrections.
  • Direct motor control — Uses NexStar passthrough commands (P) to control individual motor axes (Azimuth = axis 16, Altitude = axis 17) at fixed or variable speeds.
  • Time/Location sync — W command sends lat/lon as raw binary bytes, H command sets the date/time on the mount.
  • Backlash compensation — App-side per-axis backlash compensation on direction reversal.
  • Altitude safety limits — Configurable min/max altitude to prevent dangerous GoTo slews.
  • Autoguide rate — Set via NexStar passthrough to motor controllers (0.25x–1x sidereal).
  • Hibernate / position save — Save and restore axis positions across power cycles.
  • Speed compensation (ppm) — Correct tracking clock errors in parts per million.
  • Tracking rate control — Sidereal, lunar, and solar rates via the T command.

See Chapter 15b: NexStar/SynScan Extended Features for full documentation of these features.

The SynScan protocol specifies that software should wait up to 5 seconds for a response on position/goto commands (per the official SynScan Serial Protocol v3.3 documentation). The app uses this conservative timeout to ensure reliability.

How USB Serial Works on Android

Standard Linux serial access (/dev/ttyUSB0) is not available on Android without root. The app uses a completely different approach:

Python (telescope_bridge.py) | calls serial.write() / serial.read() v AndroidSerialPort (serial_bridge.py) - Drop-in replacement for pyserial's Serial class - Translates Python serial calls to Java method calls | Chaquopy Java interop v UsbSerialManager (Kotlin) - Android USB Host API (no root required) - Handles FTDI/CP210x/CH340/PL2303 chip protocols | USB bulk transfers v USB OTG Hardware - USB OTG cable + serial adapter - Connected to mount's serial port (RJ11/DB9/USB)

The AndroidSerialPort class is a full drop-in replacement for Python's serial.Serial. It implements read(), write(), is_open, in_waiting, and all other properties that the telescope bridge expects. Your mount doesn't know or care that it's talking to an Android phone instead of a desktop computer.

  2. Connect the serial cable from the OTG adapter to your mount
  3. Android will show a permission dialog — tap Allow
  4. In the app's Connection card:
    • Mode: USB
    • Select the detected serial port from the dropdown
    • Baud rate: 9600 (standard for most mounts)
  5. Tap Connect

Android Networking: WiFi + Cellular Simultaneously

When your phone is connected to the mount's WiFi hotspot, there is an important networking challenge: the mount's WiFi has no internet access. This means weather data, plate solving via Astrometry.net (cloud), and star database downloads would normally fail.

How the App Solves This

The app includes a Cellular Network Bridge that forces internet-bound traffic over your phone's mobile data (4G/5G) connection, even while WiFi is connected to the mount:

Mount Communication Internet Services (WiFi hotspot) (weather, plate solving, databases) | | v v WiFi Interface Cellular Interface (4G/5G) 192.168.0.1:9999 via CellularHttpClient (Kotlin) | | +------ Android Phone -------------+ (both active simultaneously)

This happens automatically. You don't need to configure anything — the app detects when WiFi has no internet and routes API calls through the cellular network. This enables:

  • Weather data refreshes via Open-Meteo API (over cellular)
  • Cloud plate solving via Astrometry.net (over cellular)
  • ASTAP database downloads from SourceForge (over cellular)
  • GPS/IP geolocation for automatic observer location (over cellular)
The cellular bridge uses Android's ConnectivityManager.requestNetwork() API with NET_CAPABILITY_INTERNET to explicitly bind HTTP requests to the cellular transport. This is a standard Android networking feature (API level 21+), not a hack.

Protocol Deep Dive

This section explains the actual commands and data formats used to communicate with your mount. Understanding these helps you interpret the Log tab and diagnose communication issues.

LX200 Protocol (OnStep and Meade mounts)

LX200 is a human-readable ASCII protocol. Every command starts with : and ends with #. Responses also end with #.

Position Polling (sent every 0.5-1 second):

App sends:    :GA#          Mount replies:   +45*30:15#       (Altitude: +45d 30' 15")
App sends:    :GZ#          Mount replies:   180*22:05#       (Azimuth: 180d 22' 05")
App sends:    :GR#          Mount replies:   05:34:31.2#      (RA: 5h 34m 31.2s)
App sends:    :GD#          Mount replies:   +22*00:52#       (Dec: +22d 00' 52")
App sends:    :D#           Mount replies:   |#  or  #        (| = slewing, empty = idle)

GoTo Sequence (when you tap a target):

Step 1:  App sends:   :Sr05:34:31#     (Set target RA to 5h 34m 31s)
         Mount replies: 1#              (1 = accepted)

Step 2:  App sends:   :Sd+22*00:52#    (Set target Dec to +22d 00' 52")
         Mount replies: 1#              (1 = accepted)

Step 3:  App sends:   :MS#             (Slew to the target)
         Mount replies: 0#              (0 = slew started successfully)

The mount now physically moves the motors. The app polls :D# until the
response changes from |# (slewing) to # (arrived).

OnStep Alt/Az GoTo (for Dobsonian mounts):

Step 1:  App sends:   :Sa+45*30:15#    (Set target altitude)
Step 2:  App sends:   :Sz180*22:05#    (Set target azimuth)
Step 3:  App sends:   :MA#             (Slew to horizontal target)

Manual Slew Commands:

:Mn#   Start moving North       :Qn#   Stop North
:Ms#   Start moving South       :Qs#   Stop South
:Me#   Start moving East        :Qe#   Stop East
:Mw#   Start moving West        :Qw#   Stop West
:Q#    Emergency stop (all axes)

Speed presets:  :RG# (Guide 0.25x)  :RC# (Center 8x)  :RM# (Find 24x)  :RS# (Max ~600x)

Tracking Rate Control (used by the real-time tracking system):

:SXTR,+0.1500#    Set RA correction rate to +0.15 arcsec/s
:SXTD,-0.0230#    Set Dec correction rate to -0.023 arcsec/s
:GXTR#             Read current RA rate  (response: +0.1500#)
:GXTD#             Read current Dec rate (response: -0.0230#)

NexStar Protocol (SynScan / Celestron)

NexStar uses compact binary commands. Commands are 1-2 characters, and positions are encoded as hexadecimal fractions of a full 360-degree circle.

Coordinate Encoding:

32-bit precision: angle = hex_value / 4294967296 * 360 degrees
Example: 1234ABCD hex = 305,441,741 / 4,294,967,296 * 360 = 25.60 degrees

16-bit precision: angle = hex_value / 65536 * 360 degrees
Example: 4000 hex = 16384 / 65536 * 360 = 90.0 degrees

Position Polling:

App sends:  z       Mount replies:  AZAZ_HEX,ALTALT_HEX#    (Alt/Az in 32-bit hex)
App sends:  e       Mount replies:  RA_HEX,DEC_HEX#         (RA/Dec in 32-bit hex)
App sends:  L       Mount replies:  0# (idle) or 1# (slewing)

GoTo Command:

App sends:  r{RA_hex32},{DEC_hex32}     RA/Dec GoTo (32-bit precision)
App sends:  b{AZ_hex32},{ALT_hex32}     Alt/Az GoTo (32-bit precision)
App sends:  s{RA_hex32},{DEC_hex32}     Sync current position

Motor Control (Passthrough):

P + chr(2) + axis + direction + rate + 0 + 0 + 0
  axis: 16 = Azimuth, 17 = Altitude
  direction: 36 = positive, 37 = negative
  rate: 0 = stop, 1-9 = fixed speeds

Variable rate (for precise tracking):
P + chr(3) + axis + direction + rate_high + rate_low + 0 + 0
  rate = arcsec_per_sec * 4, split into high/low bytes

🎯 3. Understanding the Position Display

The Position card at the top of the screen shows the telescope's current pointing position. This is the most important real-time indicator in the app.

Alt/Az Coordinates (Primary)

Since this is a Dobson alt-azimuth mount controller, Altitude and Azimuth are the native coordinate system:

CoordinateRangeDescription
Altitude-90° to +90°Angle above the horizon. 0° = horizon, 90° = zenith (directly overhead), negative = below horizon
Azimuth0° to 360°Compass bearing. 0° = North, 90° = East, 180° = South, 270° = West
These values are read directly from the telescope mount every 1-2 seconds. They represent the actual physical pointing direction of the telescope.

RA/Dec Coordinates (Derived)

Right Ascension and Declination are calculated from Alt/Az using your observer location and the current time:

CoordinateFormatDescription
RAHHh MMm SS.SsRight Ascension in hours (0-24h). Position along the celestial equator.
Dec±DD° MM' SS"Declination in degrees (-90° to +90°). Angular distance from celestial equator.
RA/Dec accuracy depends on having the correct observer latitude/longitude and system clock. Always verify your location settings in the Location tab.

Tracking Rates

The Alt Rate and Az Rate show the current tracking correction rates in arcseconds per second. These are only active when the tracking system is running.

Real-Time Slewing Updates

When the telescope is performing a GoTo slew:

  • A SLEWING badge appears next to "Position"
  • The position card border glows orange
  • An info bar shows "Telescope is slewing to target..."
  • Position coordinates update every ~1 second during slewing (faster than normal 2-second poll) so you can watch the telescope move in real-time
  • When the slew completes, the badge disappears and the final position is shown
Position Update Flow: Telescope Mount (OnStep firmware) | | :GA# (Get Altitude) every 1-2s | :GZ# (Get Azimuth) | :D# (Slew status check) v Telescope Bridge (telescope_bridge.py) | | on_altaz_update callback | Parses LX200 DMS strings -> decimal degrees v Protocol (lx200_protocol.py) | | alt_degrees, az_degrees -> derive RA/Dec | _alt_az_to_ra_dec() using LST + latitude v Web Server (_collect_state) | | JSON via /api/status (polled by JS) v Browser UI (pollStatus -> Position Card)

🌠 4. GoTo & Catalog Navigation

The GoTo tab provides catalog browsing, object search, real-time visibility calculation, and telescope slewing to any deep-sky object, star, or solar system body.

4.1 Available Catalogs

The app loads celestial object catalogs from C/C++ header files (.h) embedded in the app's data/ directory. All coordinates are stored as decimal hours (RA) and decimal degrees (Dec).

CatalogObjectsContentsSource File

4.4 GoTo Process

When you tap GoTo, the slew sequence depends on your mount protocol:

LX200 Protocol (Standard)

  1. The app converts the catalog RA/Dec to LX200 sexagesimal format strings
  2. Sends :Sr{RA}# to set the target Right Ascension
  3. Sends :Sd{Dec}# to set the target Declination
  4. Sends :MS# to start the slew
  5. The mount begins slewing — the SLEWING indicator appears
  6. Position updates show real-time progress as the telescope moves
  7. When the slew completes, the telescope reports stationary status

OnStep Alt/Az GoTo (Dobsonian Mounts)

When OnStep is detected, the app can use native Alt/Az GoTo for Dobsonian mounts, which is more accurate because it avoids the RA/Dec → Alt/Az conversion on the mount side:

  1. Convert RA/Dec to Alt/Az using the observer's latitude, longitude, and current Local Sidereal Time
  2. Send :Sa{Alt}# to set the target altitude
  3. Send :Sz{Az}# to set the target azimuth
  4. Send :MA# to slew to the horizontal coordinates
Standard LX200 protocol only supports RA/Dec GoTo (:MS#). The native Alt/Az GoTo (:MA#) is an OnStep extension and is automatically used when an OnStep mount is detected.

NexStar Protocol GoTo

For NexStar-compatible mounts:

  1. RA/Dec are encoded as 32-bit hexadecimal fractions of a full circle
  2. Send r{RA_hex32},{DEC_hex32} for RA/Dec GoTo (32-bit precision)
  3. Or send b{AZ_hex32},{ALT_hex32} for direct Alt/Az GoTo

4.8 Solar System Ephemeris Engine

Solar system objects are computed in real time using low-precision algorithms from Jean Meeus' Astronomical Algorithms:

BodyMethodAccuracy
Sun Mean longitude + equation of center (3 harmonic terms) ~1 arcmin
Moon Simplified lunar theory (Meeus Ch.47): 16 longitude terms + 10 latitude terms ~2 arcmin (~0.3°)
Planets Keplerian orbital elements (JPL/Standish 1992) + Newton-Raphson Kepler solver (20 iterations) + geocentric correction via Earth's heliocentric position ~1 arcmin

The computation pipeline for planets is:

  1. Compute Julian Date from current UTC
  2. Propagate Keplerian elements to current epoch (J2000 + rate * T)
  3. Solve Kepler's equation: E - e*sin(E) = M (Newton-Raphson)
  4. Compute heliocentric ecliptic (x,y,z) for planet and Earth
  5. Subtract to get geocentric ecliptic coordinates
  6. Rotate ecliptic -> equatorial (obliquity = 23.44°)
  7. Convert to RA/Dec
Solar warning: The Sun is included in the catalog for reference (e.g., to know where it is relative to your target). Never point a telescope at the Sun without a certified solar filter. Unfiltered solar observation will cause instant, permanent eye damage and can damage your camera sensor.

6.6 How to Interpret Telemetry Data - Practical Examples

Scenario 1: "My tracking keeps drifting in one direction"

Check the Tracking Error chart. If you see a steady upward trend in one axis:

  1. Check the Kalman v_alt/v_az - these show the estimated drift velocity
  2. Check the ML Drift Predictor - is the mean error high? The model may need more training data
  3. Check your observer location - wrong latitude causes systematic Alt/Az conversion errors

Scenario 2: "Tracking oscillates back and forth"

This often means the correction loop is overcorrecting:

  1. Check the Correction Sources weights - if one source is dominating, it may be overreacting
  2. Check Kalman Adaptive R - if very low, the filter trusts plate solves too much and amplifies noise
  3. The oscillation period may match a PEC period - check if PEC is active

Scenario 3: "RMS is good but position seems wrong"

Low RMS but wrong position usually means the initial position was wrong:

  1. Do a manual plate solve to verify the real position
  2. Check if the Kalman Initialized shows "Yes"
  3. The first plate solve establishes the reference point - if it was wrong, all subsequent corrections are offset

Scenario 4: "Plate solves are failing"

Go to the Log tab to see plate solve errors. Common causes:

  • Camera not connected or not capturing
  • Poor sky conditions (clouds, light pollution)
  • No API key set (for cloud solving mode)
  • Mount position too far off (solver can't match star patterns if pointing is wildly wrong)

⭐ 7. Plate Solving

Plate solving determines the exact sky coordinates that the telescope is pointing at by matching star patterns in a camera image against a star catalog. This is the foundation of the app's tracking system — each plate solve provides a fresh position measurement that feeds the Kalman filter and ML drift predictor.

Solver Modes

Three modes are available, selectable in Location tab > Plate Solver > Mode:

ModeDescriptionInternet Required?Speed
Auto (Recommended) Tries ASTAP local solver first. If ASTAP fails or is unavailable, automatically falls back to Astrometry.net cloud solver. Only for fallback 1-5s local, 30-60s cloud
ASTAP Local ASTAP offline solver only. Requires a downloaded star database. Will NOT fall back to cloud if solving fails. No 1-30s (with hint: 1-5s)
Astrometry.net Cloud solver only. Uploads the image to nova.astrometry.net servers for solving. Requires cellular internet. Yes (cellular) 30-60s typical
Auto mode is recommended for most users. It gives you fast offline solving when a star database is installed, with a cloud safety net when local solving fails.

ASTAP Local Solver (Offline)

ASTAP (Astrometric STAcking Program) is a professional-grade plate solver used by astrophotographers worldwide. It uses hash-based star pattern recognition against the Gaia DR3 catalog for sub-arcsecond accuracy.

How It Works

  1. The camera captures a sky image
  2. ASTAP extracts star positions from the image
  3. Star patterns are matched against the installed star database (.1476 files)
  4. On match: returns precise RA/Dec coordinates of the image center
  5. The result feeds the tracking correction pipeline (Kalman + ML)

Requirements

  • Star database: At least one database must be downloaded (see below)
  • Camera: Any connected camera (ZWO ASI, USB UVC, or phone camera)
  • No internet needed — everything runs locally on the phone

Performance Tips

  • Position hints: If the app knows your approximate pointing direction (from the mount or a previous solve), ASTAP solves in 1-5 seconds instead of 10-30 seconds
  • Correct FOV: Configure your telescope's focal length and sensor width in Optics / FOV (see section below). This lets ASTAP know exactly what image scale to expect, dramatically improving solve speed and reliability. Use D05 for most telescopes (0.6°+), W08 for phone cameras.
  • Good focus: Out-of-focus images have bloated stars that are harder to detect. Focus carefully for best results.

🔍 8. Tracking System

The real-time tracking system is the heart of TrackWise AltAz Pro. It maintains the telescope's pointing on a celestial object by continuously correcting for Earth's rotation, mechanical drift, atmospheric refraction, thermal expansion, tube flexure, and periodic gear errors — all running at 5 corrections per second.

Unlike an equatorial mount where sidereal tracking is a constant 15″/s in RA, an Alt-Az mount must vary both axes continuously. The tracking rates depend on the target's current Alt/Az position and the observer's latitude. This is why the tracking system is far more complex than a simple clock drive.

8.1 Starting & Stopping Tracking

Prerequisites

  1. Telescope is connected (WiFi or USB — see Chapter 2)
  2. A target is centred in the eyepiece/camera (via GoTo or manual slewing)
  3. A camera source is active (ZWO ASI, USB, or phone — see Chapter 9)
  4. Plate solving is working (test with a manual solve first — see Chapter 7)

Step-by-Step

  1. Navigate to the Tracking tab
  2. Tap Start Tracking
  3. The system initialises: Kalman filter resets to the current position, the PEC model is loaded from the previous session (if any), the flexure model is loaded, and all smoothing state is zeroed
  4. Corrections ramp up gradually over ~1.5 seconds (the ramp factor increases from 0 to 1.0) — this prevents an initial jolt when corrections first kick in
  5. Enable Auto Plate Solving so the camera automatically captures and solves images (target interval: every ~4 seconds)
  6. The 5 Hz control loop begins sending correction rates to the mount
  7. The real-time correction graph appears, showing individual contributions from each source

Stopping

Tap Stop Tracking. The system:

  1. Saves the trained PEC model to software_pec.json (reloaded next session)
  2. Saves the flexure model to flexure_model.json
  3. Gradually ramps down the tracking rates over ~1 second (5 steps: 100% → 80% → 60% → 40% → 20% → 0%) to avoid a mechanical jolt when the mount stops
  4. Sends a final zero rate command to guarantee a full stop
Smooth start/stop: Both ramp-up and ramp-down are designed for astrophotography — if you are in the middle of a long exposure, the telescope decelerates gently instead of jerking to a halt, which would cause star trailing on the image.
Tip: The longer you track, the better PEC and flexure models become. Models persist across sessions, so over multiple nights the system continuously improves its correction accuracy.

8.3 The 5 Hz Control Loop (Step by Step)

A dedicated background thread wakes every ~20 ms and fires a correction cycle every 200 ms (5 Hz). Each cycle executes these 9 steps:

StepOperationDetails
1 Kalman velocity Query the Kalman filter's current drift velocity estimate in Alt/Az (deg/sec). This is the smoothed, filtered drift as measured by recent plate solves.
2 ML drift prediction Query the SGD regression model for the expected drift at the current (Alt, Az) sky position. Captures learned systematic drift that depends on where the telescope is pointing.
3 Weight fusion Combine Kalman (45%) and ML (55%) velocities into a single fused drift estimate (deg/sec in Alt and Az).
4 Base sidereal rates Compute the theoretical Alt/Az tracking rates needed to follow Earth's rotation at the current position and latitude (see Section 8.4). Returns arcsec/sec.
5 Apply drift correction Subtract the fused drift from the base sidereal rates (negative feedback). If the telescope is drifting north, a southward rate component is added. Drift is converted from deg/sec to arcsec/sec (×3600).
6 Atmospheric refraction Compute Saemundsson refraction at the current altitude/temperature/pressure. The rate of change from the previous cycle is applied to the Alt tracking rate (important below ~10° altitude).
6b Thermal drift Estimate mechanical pointing shift from temperature change since alignment. Applied as a rate correction on both axes (clamped to ±5 arcsec/sec max).
7 Software PEC Feed the residual drift (what Kalman+ML could not predict) into PEC for learning. Query PEC for its Fourier-based correction and add it on top.
7b Flexure model Feed the final residual (after all other corrections) into the flexure model for learning. Query the model for its grid/parametric correction and add it on top.
8 Record for plotting Save all individual source contributions and final rates into a CorrectionRecord for the real-time stacked area chart.
9 Smooth & send to mount Apply the 4-stage output smoothing chain: EMA (alpha=0.15), start/stop ramp, rate clamp (±50″/s), and acceleration limiter (±2.5″/s per cycle). Then send to the mount (see Section 8.10 for details).

8.4 Sidereal Rate Computation for Alt-Az Mounts

For an Alt-Az mount, the sidereal tracking rates are not constant — they depend on the current altitude, azimuth, and observer latitude. The formulas are derived from the time derivatives of the Alt-Az coordinate transformation under Earth's rotation:

  dAlt/dt = omega * cos(lat) * sin(az)
  dAz/dt  = omega * (sin(lat) - tan(alt) * cos(lat) * cos(az))

  where omega = 7.2921159 × 10^-5 rad/sec (Earth's rotation rate)
Zenith singularity: Near the zenith (Alt > ~89°), the azimuth rate diverges because tan(alt) approaches infinity (field rotation). The app clamps the azimuth rate to zero when cos(alt) < 0.01. Avoid tracking objects passing directly overhead.

Performance optimisation: Latitude trig values (cos(lat), sin(lat)) are pre-computed once when the latitude is set and cached. The sidereal rate result itself is cached and only recomputed when the telescope position changes by more than 0.01° (~36 arcsec), eliminating redundant trig calls in the 5 Hz loop.

8.5 Kalman Filter (45% Weight)

The Kalman filter smooths noisy plate-solve measurements and estimates the underlying drift velocity. It is the most responsive correction source — reacting to sudden changes within a few cycles.

State Vector (4 dimensions)

  x = [alt, az, v_alt, v_az]

  alt, az:     Current position in degrees
  v_alt, v_az: Drift velocity in degrees/second

Motion Model (constant velocity)

  alt(t+dt) = alt(t) + v_alt * dt
  az(t+dt)  = az(t)  + v_az  * dt

Observation Model

Only position [alt, az] is measured from plate solving. Velocity is inferred by the filter from the sequence of position measurements.

  H = [[1, 0, 0, 0],   ← measures alt
       [0, 1, 0, 0]]   ← measures az

Noise Parameters

ParameterValueMeaning
Process noise Q (position) 0.01 deg² Allows ~36 arcsec jitter per step (models imperfect motion model)
Process noise Q (velocity) 0.001 deg²/s² Allows gradual drift changes (tuned for belt/planetary gearbox drives)
Measurement noise R 0.001 deg² ~3.6 arcsec standard deviation (matches typical ASTAP plate-solve accuracy)

Adaptive R Matrix

The AdaptiveKalmanFilter extends the standard filter by dynamically adjusting the measurement noise R based on observed residuals. If plate-solve scatter increases (e.g. due to clouds or vibration), R increases automatically, causing the filter to trust measurements less and rely more on the motion model. This prevents noisy measurements from destabilising the velocity estimate.

The adaptation runs every few updates by computing the RMS of recent residuals (innovations) and scaling R proportionally.

The ML drift predictor learns systematic position-dependent drift patterns using online Stochastic Gradient Descent (SGD) linear regression. Unlike the Kalman filter which reacts to observed drift, the ML model predicts expected drift based on the telescope's current sky position.

Learning

  • Online SGD: Each new drift sample triggers a single SGD weight update (learning rate: 0.01, L2 regularization: 0.001)
  • Normalization: Alt and Az are z-score normalized (Alt: mean=45°, std=30°; Az: mean=180°, std=100°) to keep feature magnitudes balanced
  • Sample buffer: Up to 5,000 samples are retained in memory
  • Persistence: Model weights are saved to drift_model.json and reloaded on restart so learned patterns survive across sessions

Output Smoothing

The raw dot-product predictions from the regression model can jump abruptly when the telescope crosses into sky regions with different learned weight contributions. To prevent these jumps from reaching the mount, the ML predictor applies an Exponential Moving Average (alpha = 0.2) on its output:

  smooth_alt = 0.2 * raw_prediction + 0.8 * previous_smooth_alt
  smooth_az  = 0.2 * raw_prediction + 0.8 * previous_smooth_az

This is separate from the main pipeline EMA smoother (Section 8.10) and provides an additional layer of protection against sudden rate changes.

Tip: The ML model improves significantly after 10-15 minutes of tracking as it accumulates enough samples to learn the position-dependent drift pattern of your specific mount. Over multiple sessions, the saved model provides immediately accurate predictions.

8.7 Periodic Error Correction (PEC) — Complete Guide

What is Periodic Error?
Every telescope mount has small mechanical imperfections — slightly uneven gear teeth, belt stretch, bearing wobble. These imperfections cause the mount to speed up and slow down in a repeating pattern as the gears rotate. This creates a rhythmic back-and-forth tracking error called periodic error. PEC (Periodic Error Correction) learns this pattern and cancels it by adjusting the motor speed in real time.

Two Independent PEC Systems

Software PEC — How It Works (No Sensor Needed)

Software PEC is entirely sensorless. It does not read from any hardware PEC sensor. Instead, it uses your camera and plate solving to measure the actual tracking error, then uses math (FFT + Fourier analysis) to find the repeating patterns and cancel them:

  Step 1: Camera captures sky image (every ~4 seconds)
          |
  Step 2: ASTAP plate-solves the image -> precise RA/Dec position
          |
  Step 3: Kalman filter + ML predictor correct what they can
          |
  Step 4: The LEFTOVER error (what Kalman+ML could NOT predict)
          is fed to the PEC engine
          |
  Step 5: PEC runs FFT analysis on accumulated residuals
          -> discovers periodic frequencies (gear mesh, worm period, etc.)
          -> fits a multi-harmonic Fourier series to each period
          |
  Step 6: At every tracking tick (5 Hz), PEC evaluates its Fourier model
          -> predicts what the periodic error will be RIGHT NOW
          -> applies the NEGATIVE (opposite) as a motor speed correction
          |
  Result: The periodic error is cancelled in real time
Key insight: PEC only receives the residual error that Kalman filtering and ML prediction could not explain. This isolates the pure periodic mechanical component. The system automatically discovers the frequencies — it does not need to know your gear ratios in advance.

Mount-Side PEC — How It Works (Requires Worm Sensor)

Mount-side PEC is the traditional approach used in OnStep/OnStepX mounts. It runs entirely on the mount's controller board and uses a physical worm index sensor (a small magnet + Hall sensor on the worm gear) to know exactly where in the worm rotation cycle the mount currently is:

  Step 1: You guide the mount well for one complete worm cycle
          (typically 5-10 minutes depending on worm gear ratio)
          |
  Step 2: Press "Record" -> mount firmware records the corrections
          needed at each position in the worm cycle
          |
  Step 3: Press "Save NV" -> corrections saved to EEPROM
          |
  Step 4: Press "Play" -> mount applies corrections automatically
          on every subsequent worm revolution
          |
  The worm index sensor keeps the correction synchronized
  with the physical worm position
Can I use both at the same time? Yes. Mount-side PEC corrects the primary worm gear period at high frequency on the controller. Software PEC then detects and corrects any residual periodic errors that mount PEC missed — such as bearing wobble, belt tooth spacing, or higher-order harmonics. Using both together gives the best possible correction.

Which One Should I Use?

  • No worm sensor on your mount? → Use Software PEC (it works with any mount, any drive type)
  • OnStep/OnStepX with worm sensor? → Use Mount-Side PEC for the worm period, plus Software PEC for residual errors
  • Non-worm drive (planetary, belt, harmonic)? → Software PEC only (mount PEC requires a worm sensor)
  • Direct drive? → Software PEC (very low periodic error, but bearing irregularities may exist)

Software PEC — Technical Details

Software PEC automatically detects and corrects repeating mechanical errors caused by gears, belts, and bearings — without requiring physical encoders or sensors.

Analysis Pipeline

  1. Collect: Each tracking cycle, the post-Kalman/ML residual error is recorded. These residuals isolate the periodic component that the primary correction sources cannot predict.
  2. Detect: Every 30-60 seconds, the residuals are resampled onto a uniform time grid, detrended, windowed (Hann), and passed through an FFT to find dominant periodicities.
  3. Fit: For each detected period, a multi-harmonic Fourier series is fit via least-squares regression: 
      correction(t) = SUM [a_n * cos(2*pi*n*f0*t) + b_n * sin(2*pi*n*f0*t)]
  4. Correct: At each 5 Hz cycle, the Fourier model evaluates the predicted error at the current time and returns the negative (to cancel it).

Drive Type Presets

Different drive systems have different periodic error characteristics. Configure your drive type in Settings for optimal PEC performance:

Spectral Analysis Details

  • A Hann window is applied before the FFT to suppress spectral leakage from non-integer cycle counts
  • A 3× guard band (Hann main lobe width) is used as the exclusion zone around each detected peak
  • Peaks are validated as local maxima (not just threshold crossings) to avoid counting sidelobes
  • Detected peaks are checked against existing peaks to reject harmonics of already-detected fundamentals (ratio close to integer = skip)

Coefficient Updates

When re-analysing (every 30-60s depending on drive type), new Fourier coefficients are blended with existing ones via Exponential Moving Average (EMA):

  coeff_new = coeff_old * (1 - lr) + coeff_measured * lr

This avoids abrupt jumps in the correction signal when the model updates mid-observation.

Persistence: The learned PEC model (detected periods + Fourier coefficients) is saved to software_pec.json and reloaded automatically on the next session. The model immediately begins correcting; new data gradually re-locks the phase and refines coefficients through EMA blending.

8.8 Environmental Corrections

Atmospheric Refraction (Altitude Only)

The atmosphere bends light upward, making objects appear higher than their true geometric position. The refraction amount depends on altitude, temperature, and pressure.

Altitude RangeFormulaTypical Refraction
> 10° Saemundsson: R = 1.02 / tan(alt) arcmin ~5 arcsec at 45°, ~1 arcsec at 75°
≤ 10° Extended: R = 1.02/tan(alt) - 0.001927/tan(alt)^3 ~5.3 arcmin at 1° (very large!)

Both formulas are scaled by temperature and pressure factors:

  temp_factor = 283.15 / (273.15 + T_celsius)     [reference: 10°C]
  press_factor = P_hpa / 1013.25                   [reference: sea level]

The tracking system computes refraction at the current position each cycle and applies the rate of change (delta from the previous cycle divided by dt) as an additional Alt tracking rate. The refraction rate is clamped to ±5 arcsec/sec (matching the thermal drift clamp) to prevent sudden jumps when atmospheric conditions change rapidly or when the first refraction reading arrives.

Near-horizon warning: Below ~5° altitude, refraction changes rapidly and can exceed the correction system's range. The atmosphere is also highly variable near the horizon (turbulence, temperature inversions), making accurate correction difficult. Avoid long-exposure tracking below 10° altitude.

Thermal Drift (Both Axes)

Temperature changes cause the telescope tube and mount structure to expand or contract, shifting the optical axis. The correction model is:

  correction_alt = (T_current - T_baseline) * thermal_coefficient
  correction_az  = correction_alt * 0.5
ParameterDefaultDescription
Thermal coefficient 0.1 arcsec/°C Sensitivity of pointing to temperature change (typical range: 0.05-0.3 depending on tube material and length)
Baseline temperature Auto (first reading) Set automatically from the first temperature reading when tracking starts
Rate clamp ±5 arcsec/sec Maximum thermal correction rate to prevent jumps when temperature data first arrives

Altitude is the primary affected axis (tube expansion changes the balance point and flexure). The azimuth correction is empirically set to half the altitude correction for typical Dobson mounts.

8.9 Flexure Model (Sky Map Learning)

The flexure model learns the relationship between telescope pointing direction, temperature, and mechanical tube flexure by accumulating tracking residuals over time. It captures effects that the analytical formulas (refraction, thermal) and the ML predictor cannot fully model.

Why Flexure Matters

  • At low altitudes, gravity acts perpendicular to the tube, creating maximum bending (optical axis tilts downward)
  • At the zenith, gravity acts along the tube — minimal bending
  • Azimuth matters because the tube is asymmetric: the focuser, finder, camera, and cables shift the flexure direction
  • Temperature changes alter tube stiffness (thermal expansion, material modulus), modifying flexure magnitude

Dual-Representation Architecture

The model maintains two representations that are blended:

1. Grid Map (18 × 36 = 648 cells)

  • 18 altitude bins (5° resolution) × 36 azimuth bins (10° resolution)
  • Each cell stores the average residual correction in Alt and Az (arcsec/sec)
  • Updated via EMA (learning rate: 0.05 = 5% per new sample)
  • Bilinear interpolation between cells for smooth corrections
  • Provides high-resolution spatial structure of the flexure

2. Parametric Model (8 coefficients)

  flexure_alt = A*cos(alt) + B*cos(alt)*cos(az) + C*cos(alt)*sin(az)
              + D*cos(alt)*dT + E*dT

  flexure_az  = F*sin(az)*cos(alt) + G*cos(az)*cos(alt) + H*dT
  • A*cos(alt): Gravitational flexure (maximum at horizon, zero at zenith)
  • B, C: Azimuthal asymmetry from off-axis mass (focuser, camera)
  • D: Thermal-gravity interaction (flexure changes with temperature)
  • E, H: Pure thermal drift (temperature-only component)
  • Updated via SGD with L2 regularization every 10 samples
  • Provides predictions in un-observed sky regions by extrapolation

Blending rule: In well-observed grid cells (~20+ samples), the grid correction dominates. In poorly-observed or un-visited cells, the parametric model fills in via smooth blending based on a confidence score.

Persistence: The entire model (grid + parametric coefficients) is saved to flexure_model.json. Over multiple observing sessions, the model accumulates knowledge of your specific telescope's flexure patterns across the entire visible sky.

8.10 Rate Delivery to the Mount

After computing the final Alt/Az rates, the tracking system delivers them to the mount using one of two paths:

Path A: NexStar Variable-Rate Passthrough (Preferred for Alt-Az)

If a NexStar-compatible mount is detected, Alt/Az rates are sent directly via the NexStar passthrough 'P' command. This is more accurate because it avoids the Jacobian approximation entirely — the mount receives exactly the Alt/Az rates the tracking system computed.

Path B: OnStepX SXTR/SXTD (Equatorial Rate Offsets)

For OnStepX and LX200-compatible mounts, the Alt/Az rates must be converted to equatorial RA/Dec rates because the OnStepX SXTR/SXTD commands expect equatorial coordinates:

  • :SXTR,n.n# — Set tracking rate offset for RA axis in arcsec per sidereal second. The mount adds this to the currently active base tracking rate (sidereal, lunar, solar, or king).
  • :SXTD,n.n# — Set tracking rate offset for Dec axis in arcsec per sidereal second.

Base Rate Awareness

SXTR is an offset from the mount's current base tracking rate. The app tracks which rate mode is active and subtracts the correct base rate before sending:

Mount ModeCommandBase RA Rate (arcsec/SI-sec)
Sidereal:TQ#15.041″/s
Lunar:TL#14.685″/s
Solar:TS#15.000″/s
King:TK#15.037″/s

When you change the tracking rate in the UI, the app sends the standard :TQ#/:TL#/:TS#/:TK# command to the mount and updates the correction pipeline so the SXTR offset is computed relative to the new base rate.

Sidereal Second Unit Conversion

The internal pipeline computes rates in arcsec per SI second, but OnStepX SXTR/SXTD expect arcsec per sidereal second. One sidereal day = 86,164.0905 SI seconds, so one SI second = 1.002738 sidereal seconds. The app multiplies both offsets by this factor before sending:

  SXTR_value = (RA_rate_SI - base_rate) × 1.002738
  SXTD_value = Dec_rate_SI × 1.002738

Alt/Az → RA/Dec Jacobian

The conversion uses the local Jacobian of the (Alt/Az) → (RA/Dec) transformation at the current position:

  dRA/dAlt  = -cos(lat)*sin(az) / (cos(dec)*cos(alt))
  dRA/dAz   = (sin(lat)*cos(alt) - cos(lat)*sin(alt)*cos(az)) / (cos(dec)*cos(alt))
  dDec/dAlt = (sin(lat)*cos(alt) - cos(lat)*sin(alt)*cos(az)) / cos(dec)
  dDec/dAz  = cos(lat)*cos(alt)*sin(az) / cos(dec)
Singularity: Near the celestial pole (cos(dec) → 0) or zenith (cos(alt) → 0), the Jacobian is singular. The app returns a safe fallback (0, alt_rate) in these cases.

4-Stage Output Smoothing Chain

Before the final rates reach the mount motors, they pass through a 4-stage smoothing pipeline designed for ultra-smooth corrections that are nearly invisible to the naked eye during long-exposure astrophotography:

StageParameterWhat It Does
1. EMA Smoother alpha = 0.15 Exponential Moving Average debounces plate-solve jitter. Lower alpha = heavier smoothing.
smooth = 0.15 * new + 0.85 * prev
2. Start/Stop Ramp ~1.5s ramp-up Multiplies rates by a ramp factor that increases from 0.0 to 1.0 over ~1.5 seconds after tracking starts (0.13 per 200ms cycle). Prevents the initial correction jolt. On stop, rates ramp down over ~1 second (5 steps: 100%→0%).
3. Max Rate Clamp ±50″/s Hard ceiling on both Alt and Az rates. Prevents runaway corrections from ever reaching the mount motors, regardless of upstream errors.
4. Acceleration Limiter ±2.5″/s per cycle Limits how much the sent rate can change between consecutive 200ms cycles. At 5Hz this allows a maximum acceleration of 12.5″/s². Ensures corrections change imperceptibly smoothly.

Why 4 Stages?

Each stage addresses a different failure mode:

  • EMA handles high-frequency noise (plate-solve jitter, sensor noise)
  • Ramp handles transient startup/shutdown events
  • Rate clamp is a safety net against algorithmic runaway (e.g. a corrupted plate solve returning extreme coordinates)
  • Acceleration limiter guarantees the rate of change of the rate stays imperceptibly small — the most critical factor for astrophotography smoothness
Astrophotography result: With all 4 stages active, mount corrections are so gradual they are nearly invisible to the naked eye. During long exposures (30s+), correction changes are spread across multiple cycles, preventing the sharp “step” artifacts that cause star trailing.

8.12 Tracking Quality Monitor

The LongTermTrackingMonitor continuously evaluates tracking health and classifies it into quality levels:

Automatic Alerts

The monitor fires alerts when:

  • Cumulative drift exceeds 60 arcsec (1 arcmin) — object has wandered significantly
  • RMS error exceeds 10 arcsec — generally poor tracking
  • Plate-solve success rate drops below 70% (after 50 solves) — Kalman filter is mostly predicting, not correcting

Every 1 hour, the monitor recommends a Kalman filter reset to flush accumulated systematic bias. The cumulative drift counters are zeroed and a fresh convergence begins. This prevents long-term drift accumulation from small systematic errors.

🌥️ 11. Location & Weather

Accurate observer location and real-time weather data are essential for coordinate conversions, atmospheric refraction corrections, dew risk assessment, and observing conditions evaluation.

11.1 Why Location Matters

Your latitude and longitude are used throughout the app:

SystemHow Location Is Used
Coordinate Conversion Alt/Az ↔ RA/Dec transformations require the observer's latitude. The _radec_to_altaz() function in the tracking system uses latitude for the spherical rotation matrix.
Local Sidereal Time LST = GST + longitude. The LST determines which RA is on the local meridian, and is essential for all Alt/Az ↔ RA/Dec conversions.
Sidereal Tracking Rates The Alt/Az sidereal tracking formulas (dAlt/dt = omega*cos(lat)*sin(az)) depend directly on latitude. Incorrect latitude means incorrect base tracking rates.
Object Visibility Determining whether objects are above/below the horizon requires knowing the observer's position on Earth.
Auto-Alignment Star selection uses Alt/Az visibility computed from your location to pick stars above the horizon.
Weather Data The Open-Meteo API requires GPS coordinates to return local weather conditions.
Important: If your latitude is wrong by even 1°, the sidereal tracking rates will be slightly off, the Alt/Az ↔ RA/Dec conversions will have systematic errors, and GoTo accuracy will degrade. Always verify your location is correct before an observing session.

11.4 Cellular Network Routing (Android)

When your phone is connected to the telescope's WiFi hotspot (which has no internet), weather API calls would normally fail because Android routes all traffic through the active WiFi connection.

The app solves this with the CellularHttpClient — a Kotlin class that explicitly routes HTTP requests through the 4G/5G cellular network:

  Normal Android:
    App --> WiFi (telescope hotspot, no internet) --> FAIL

  TrackWise AltAz Pro:
    App --> CellularHttpClient --> ConnectivityManager.requestNetwork()
        --> Binds to TRANSPORT_CELLULAR --> 4G/5G internet --> SUCCESS

How It Works

  1. At app startup, CellularHttpClient.requestCellularNetwork() asks Android's ConnectivityManager for a cellular network reference
  2. A NetworkCallback monitors cellular availability and caches the Network object
  3. When the weather service needs to fetch data, it first tries cellular_get() via the Python bridge
  4. The Kotlin client opens the HTTP connection bound to the cellular network: cellularNetwork.openConnection(url)
  5. This bypasses the WiFi entirely — the request goes over 4G/5G
  6. If cellular is unavailable (no SIM, airplane mode, etc.), it falls back to the default network (which works on desktop or normal WiFi)
Both WiFi and cellular are active simultaneously. The telescope connection uses WiFi (local network), while weather and plate-solve cloud requests use cellular data. Android supports this dual-network configuration natively through the ConnectivityManager API.

11.5 Dew Point Calculation

The dew point is computed locally using the Magnus formula (valid for -45°C to +60°C):

  a = 17.27,  b = 237.7

  alpha = (a * T) / (b + T) + ln(humidity / 100)

  dew_point = (b * alpha) / (a - alpha)

Why dew point matters for astronomy:

  • When the ambient temperature drops to within 2-3°C of the dew point, moisture condenses on optical surfaces
  • Dew on the primary mirror, corrector plate, or eyepiece ruins observations instantly
  • The app displays the dew margin (temperature minus dew point) and warns when it's dangerously low

🔧 12. Auto-Alignment System - Complete Guide

The alignment system is one of the most powerful features in TrackWise AltAz Pro. It supports two modes: Automatic (plate-solver centers each star and syncs automatically) and Manual (you center each star visually using directional slew controls and confirm with Sync). Both modes automatically select the brightest visible stars spread across the sky and slew to each one. After all stars are aligned, GoTo accuracy is significantly improved across the entire sky.

12.1 What Auto-Alignment Does

When you buy a new GoTo mount or start a new observing session, the mount's internal pointing model has errors. Pointing the mount at M31 might land you 1-2 degrees away from the actual galaxy. The auto-alignment system fixes this by teaching the mount where stars actually are vs. where it thinks they are.

The procedure works in 5 phases:

Phase 1: HOME Phase 2: SELECT Phase 3: ALIGN Slew to home Pick N bright For each star: position stars across GoTo -> Solve -> (known reference) the sky Recenter -> Sync Phase 4: VERIFY Phase 5: COMPLETE Revisit first 2 Report results stars to check (N/M aligned OK) model convergence

Phase 1: Homing

The telescope is sent to its home position (:hF# command for LX200, or equivalent for NexStar). This establishes a known starting reference point. If homing fails (some mounts don't support it), the procedure continues from the current position.

Phase 2: Star Selection

The system loads a catalog of named bright stars (from the stars.h catalog file containing hundreds of named stars with visual magnitudes) and applies a sophisticated selection algorithm designed for optimal sky coverage. See Section 12.3 for full details on how stars are selected.

Phase 3: Per-Star Alignment (the core loop)

For each selected star, a multi-step iterative procedure runs:

  1. GoTo — Slew the mount to the star's catalog RA/Dec coordinates
  2. Wait for slew — Monitor the :D# slew status flag until the mount reports idle, then wait an additional 2 seconds for mechanical vibrations to settle
  3. Plate solve — Capture a sky image and run ASTAP (or cloud solver) to determine the actual RA/Dec the telescope is pointing at
  4. Measure error — Calculate the angular separation between the solved position and the target star (in arcseconds, using the Haversine formula)
  5. Decision:
    • If error ≤ 60 arcsec (1 arcminute): proceed to confirmation solve
    • If error > 60 arcsec: compute a correction offset and re-GoTo with the corrected position
  6. Confirmation — When within threshold, a second plate solve confirms the position. If this also passes (≤ 60"), the mount is synced
  7. Sync — Sends :Sr{RA}# + :Sd{Dec}# + :CM# (LX200 sync) to tell the mount "you are actually pointing HERE." This updates the mount's internal pointing model.

Each star gets up to 5 attempts to center. If plate solving fails or the star can't be centered after 5 attempts, it is marked as "failed" and the procedure moves to the next star.

Phase 4: Verification Pass

After all stars are aligned, the system revisits the first 2 aligned stars. This is critical because:

  • The first stars were aligned before the mount had many sync points, so the model was still crude
  • Revisiting them checks whether the accumulated model has improved their accuracy
  • If a revisited star has error > 2x threshold (120") but < 5x threshold (300"), it is re-synced
  • If error is > 5x threshold (300"), the verification notes it as a problem but does not re-sync (the error is too large for a reliable correction)

Phase 5: Completion

A summary reports how many stars were successfully aligned out of the total attempted. For example: "Alignment complete: 5/6 stars aligned successfully."

12.3 Star Selection Algorithm (How Stars Are Chosen)

The star selection is not simply "pick the N brightest." It uses a sophisticated 2D grid algorithm designed to maximize both azimuth AND altitude coverage, which is critical for accurate pointing model calibration.

Why Sky Coverage Matters

  • Azimuth coverage: If all alignment stars are in the south, the mount has no calibration data for the north. GoTo accuracy degrades in un-sampled directions.
  • Altitude coverage: Tube flexure (gravitational bending) changes significantly with altitude. Stars at 20° altitude experience very different flexure than stars at 70°. Without altitude diversity, the pointing model cannot account for this.
  • Atmospheric refraction: Varies strongly with altitude. Stars near the horizon appear shifted upward by up to 0.5 degrees.

Selection Steps

  1. Load catalog — Parse the stars.h file containing hundreds of named stars with visual magnitudes (format: magnitude * 100 as integer, e.g., 76 = mag 0.76). Stars are sorted by brightness (lowest magnitude = brightest).
  2. Compute visibility — For every star in the catalog, compute its current Alt/Az using the observer's latitude/longitude and Local Sidereal Time. Filter out any star below the minimum altitude threshold (15 degrees by default). Stars too close to the horizon are difficult to plate solve (atmospheric extinction, thick air mass).
  3. Build a 2D grid — Divide the sky into:
    • Azimuth sectors: Up to min(N, 12) sectors, giving uniform angular spacing (e.g., 6 stars = 6 sectors of 60° each)
    • Altitude bands: 3 fixed bands for different flexure regimes:
      • Low: 15° - 35° (heavy flexure, strong refraction)
      • Mid: 35° - 60° (moderate flexure)
      • High: 60° - 90° (minimal flexure, near zenith)
    Each star is placed into the (sector, band) cell matching its current position.
  4. Phase 1: Grid selection — Pick the brightest star from each (sector, band) cell, cycling through altitude bands first to ensure altitude diversity. This guarantees at least one star per altitude regime (if available).
  5. Phase 2: Fill underrepresented bands — If some altitude bands have fewer stars than others, fill from the remaining brightest candidates, preferring underrepresented bands.
  6. Phase 3: Final fill — Fill any remaining slots with the overall brightest unused visible stars.
  7. Optimize slew order — Reorder the selected stars for minimum total slew distance using a nearest-neighbour heuristic on (Alt, Az) angular distance (with azimuth wrap-around handling). This reduces total alignment time compared to a simple azimuth sort.

The Re-centering Correction

When the plate solve shows the telescope is NOT pointing at the target, the system computes a correction:

  ra_correction  = target_RA  - solved_RA     (how far off in RA)
  dec_correction = target_Dec - solved_Dec     (how far off in Dec)

  corrected_RA  = target_RA  + ra_correction   (overshoot by the error amount)
  corrected_Dec = target_Dec + dec_correction   (overshoot by the error amount)

  Then: GoTo(corrected_RA, corrected_Dec)

This effectively applies a double correction: if the mount undershot by 30 arcminutes, the next GoTo aims 30 arcminutes past the target, hoping the same systematic error will place it right on the star.

12.7 Tips for Best Results

  1. Do alignment early — Run auto-alignment at the start of your session, before the sky has rotated too far. Stars near the horizon at session start may have set by the time the alignment reaches them.
  2. Use 6-9 stars minimum — 3 stars gives a basic model but can't correct for flexure across the sky. 6+ stars provide altitude diversity that characterizes tube flexure.
  3. Ensure good focus — Out-of-focus stars produce bloated images that ASTAP struggles to solve. Use the Camera tab to achieve sharp focus before starting alignment.
  4. Dark sky preferred — Plate solving works best in dark skies where many stars are visible. In light-polluted sites, increase camera exposure or gain.
  5. Let the mount warm up — Temperature changes cause mechanical flexure. If you just brought the scope outside, wait 15-30 minutes for it to reach thermal equilibrium before aligning.
  6. Don't touch the mount during alignment — Any physical bump invalidates the sync data. Let the procedure run hands-free.
  7. Check the log — The Log tab shows detailed messages for every alignment step. If a star fails, the log tells you exactly why (solve failure, timeout, mount error, etc.).
  8. Re-align if you move the mount — If you relocate the tripod, bump the mount, or change the optical train (add/remove camera, eyepiece, filter), the previous alignment is no longer valid.
How alignment interacts with tracking: After a successful auto-alignment, the mount's internal pointing model is improved. This means GoTo commands will be more accurate. The real-time tracking system (Kalman + ML + PEC) provides additional corrections on top of the alignment model for sub-arcsecond accuracy during tracking. The two systems are complementary: alignment fixes the static pointing model, tracking fixes the dynamic drift.

12.8 Technical Details: How the Mount Sync Works

When the auto-alignment system syncs a star, it sends these commands to the mount:

1. :Sr{RA}#      Set the "target" RA to the star's known catalog position
2. :Sd{Dec}#     Set the "target" Dec to the star's known catalog position
3. :CM#          SYNC: tell the mount "your current actual position is this target"

This is also sent via the hardware bridge (if connected):
4. bridge.send_command(":Sr{RA}#")
5. bridge.send_command(":Sd{Dec}#")
6. bridge.send_command(":CM#")

The mount receives these sync commands and updates its internal pointing model. For OnStep mounts, each sync point is added to a multi-star alignment model (up to 100+ points). OnStep then uses these points to interpolate corrections across the entire sky via a least-squares best-fit algorithm. For NexStar/SynScan mounts, the sync updates the hand controller's alignment database (typically supports 1-star or 2-star models in firmware, with additional corrections in software).

Angular Separation Formula

The system uses the Haversine formula to compute the angular distance between the plate-solve result and the target star:

  cos(sep) = sin(dec1) * sin(dec2)
           + cos(dec1) * cos(dec2) * cos(ra1 - ra2)

  separation_arcsec = acos(cos(sep)) * 3600    (degrees to arcseconds)

RA values are converted from hours to radians (RA * 15 degrees) before the computation.

13.3 Crash Recovery System

The crash recovery system protects against unexpected shutdowns (power loss, app crash, OS kill). It works by saving periodic checkpoints of the application state.

How It Works

  1. Checkpoint file: .telescope_checkpoint.json is written to disk every 30 seconds
  2. Contents saved:
    • Current telescope position (RA, Dec, Alt, Az)
    • Tracking state (running, target, correction rates)
    • Kalman filter internal state (state vector, covariance matrix)
    • ML model weights
    • Session statistics (solve count, success rate, RMS)
    • Connection info (port, baud rate, protocol)
    • Clean shutdown flag
  3. On clean shutdown: The checkpoint is updated with "clean_shutdown": true
  4. On next startup: If the checkpoint shows "clean_shutdown": false, the app detects an unclean shutdown and can restore the previous state

Recovery Behaviour

Shutdown TypeWhat Happens on Restart
Clean shutdown (Stop button, app exit)Normal startup. Models and config are saved normally.
Crash / force killDetects unclean shutdown. Loads last checkpoint position and Kalman state. ML model and PEC data are preserved.
Power lossSame as crash. The checkpoint from the last 30-second cycle is used. Maximum data loss: 30 seconds of tracking data.
The crash recovery system is especially important on Raspberry Pi, where SD card corruption or power interruptions are more common. Using a UPS (Uninterruptible Power Supply) for your Pi is strongly recommended for long imaging sessions.

15b.2 Backlash Compensation

NexStar mounts have mechanical gear backlash (play) that causes a brief dead zone when reversing direction. The app provides per-axis backlash compensation that injects a corrective pulse on direction reversal to take up the slack before the visible slew starts.

How It Works

  1. Set the backlash value (in arcseconds) for each axis: RA/Azm and Dec/Alt.
  2. When you reverse direction on an axis (e.g., slew East then West), the app automatically sends a brief corrective motor pulse in the new direction.
  3. The pulse duration is calculated from backlash_arcsec / slew_rate, capped at 2 seconds.
  4. After the compensation pulse, the normal slew continues.

Setting Backlash

Go to Settings > Backlash and enter the arcsecond values for each axis. Typical values are 10-60 arcseconds depending on your mount's gearing. Set to 0 to disable compensation for an axis.

To measure your mount's backlash: center a bright star, slew East for a few seconds, then reverse to West and time how long until the star starts moving. Multiply by the approximate slew rate for the backlash in arcseconds.
Unlike OnStep backlash (which is firmware-side using :$BR# / :$BD# commands), NexStar backlash is app-side only. The values are stored in the app and applied by injecting motor passthrough commands on direction reversal.

The default 0.5x sidereal (7.5 "/s) is suitable for most autoguider setups. Use 0.25x for very stable mounts or long focal lengths, and 1x for mounts with significant periodic error.

Protocol Detail

The passthrough command format is: P + chr(2) + chr(axis) + chr(0x46) + chr(rate_pct) + chr(0) + chr(0) + chr(0), where rate_pct is the guide rate as a percentage of sidereal (1-99).

15b.5 Hibernate / Position Save

The hibernate feature saves the current mount axis positions so they can be restored after a power cycle, avoiding a full re-alignment.

Workflow

  1. Before power-off: Click Save Position. The app reads the current Alt/Az position from the mount and stores it both in memory and on disk (via the config manager).
  2. Power off the mount. Leave the mount mechanically undisturbed.
  3. After power-on: Connect to the mount, then click Restore Position. The app issues a GoTo command to the saved Alt/Az position, re-establishing the mount's coordinate reference.
Hibernate only works if the mount is not physically moved between power cycles. If the mount was bumped or repositioned, restore will point to the wrong location and you should perform a fresh alignment instead.
The saved position persists across app restarts via the configuration file (nexstar.hibernate_azm and nexstar.hibernate_alt keys).

15b.6 Speed Compensation (ppm)

Speed compensation corrects for tracking clock frequency errors in the mount's motor controller. If your mount tracks slightly too fast or too slow (visible as consistent drift in one direction), you can apply a parts-per-million correction.

Usage

ValueEffect
+5.0 ppmSpeeds up tracking by 5 parts per million
0.0 ppmNo correction (default)
-3.5 ppmSlows down tracking by 3.5 parts per million

The correction is applied app-side by adjusting the effective tracking rate. The tracking controller reads the ppm value when computing correction rates sent to the mount via the variable-rate passthrough commands.

To determine the correct ppm value: track a star for 10+ minutes without corrections, measure the accumulated drift in arcseconds, and compute: ppm = (drift_arcsec / elapsed_sec) / 15.041 * 1e6. A positive result means the mount is too slow (apply +ppm to speed up).

16.9 General Diagnostic Tips

  • Check the Log tab first. Every system logs its activity with timestamps, severity levels, and detailed messages. Filter by keyword to find relevant entries.
  • Look for colour-coded messages: Red = errors/failures, Yellow = warnings, Green = success, Blue = info
  • Use the tracking stats panel to see Kalman RMS, ML samples, PEC periods, solve success rate, flexure coverage, and other diagnostics in real time
  • Export logs if you need to report an issue. The Log tab supports copying entries to the clipboard.
  • Restart tracking if correction quality degrades suddenly. Stopping and restarting tracking reinitialises the Kalman filter and reloads the PEC/flexure models.
