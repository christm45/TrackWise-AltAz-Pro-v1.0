Session Summary -- Bug Fixes & Android Hardening

Task 1: SystemExit:1 crash when launching without ADB (3 root causes)
Root cause analysis: The app launched fine with USB ADB debug connected but crashed
with "SystemExit:1" after cable removal. Three compounding issues:

1. Missing foreground service type on Android 14 (PRIMARY CAUSE)
   - TelescopeService.kt called startForeground(id, notification) without the required
     foregroundServiceType parameter. Android 14 (API 34, targetSdk) throws
     MissingForegroundServiceTypeException without it.
   - Fix: Added ServiceInfo.FOREGROUND_SERVICE_TYPE_CONNECTED_DEVICE on API 34+ with
     Build.VERSION.SDK_INT check, backward-compatible else branch for older APIs.

2. No error handling in Python startup
   - start_server() in android_bridge/main.py had zero try/except wrapping.
     Any SystemExit from Flask/werkzeug/numpy/scipy propagated as raw PyException.
   - Fix: Split into start_server() wrapper + _start_server_inner(). Wrapper catches
     SystemExit (converts to RuntimeError with descriptive message) and BaseException
     (logs full traceback before re-raising).

3. Port 8080 stuck from previous crash
   - When app was force-killed (cable pull, ANR, OOM), old Python process could hold
     the TCP port. Next launch: make_server() fails "Address already in use".
   - Fix: Added _ensure_port_free(port) utility in main.py -- probes port with
     SO_REUSEADDR, retries up to 3 seconds. Also patched web_server.py
     TelescopeWebServer.start() to force SO_REUSEADDR on werkzeug's BaseWSGIServer
     via monkey-patched server_bind().

Additional hardening:

- TelescopeService.kt: Improved error messages in catch block -- detects SystemExit,

Task 1: Position Display Not Reflecting True Telescope Position (3 root causes)

Root cause analysis: After connecting to a telescope (real or simulator), the
Position card at the top of the screen showed default/stale values instead of
the mount's actual pointing position. Three compounding bugs:

1. on_altaz_update callback NEVER WIRED (PRIMARY CAUSE)
   - telescope_bridge._read_loop() polls the mount with :GA#/:GZ# every 2 seconds
     and fires self.on_altaz_update(alt_response, az_response).
   - HEADLESS_SERVER.py._toggle_usb_telescope() called bridge.connect() but NEVER
     set bridge.on_altaz_update to a callback function.
   - Result: Mount position data was received from hardware but discarded -- it
     never reached self.protocol or the web UI.
   - Fix: Added _wire_position_callback(bridge) method to HEADLESS_SERVER.py that
     creates a callback parsing raw LX200 DMS strings (e.g., "+45*30:15#") to
     decimal degrees using new _parse_lx200_dms() static method, then writes:
       * self.protocol.alt_degrees / az_degrees (direct from mount)
       * self.protocol.ra_hours / dec_degrees (derived via _alt_az_to_ra_dec)
       * self.protocol.is_slewing (from slew status flag)
   - Wired BEFORE connect in _toggle_usb_telescope() and _toggle_simulator()

2. Position NOT READ during slewing
   - telescope_bridge._read_loop() checked :D# for slew status. If is_slewing=True,
     it SKIPPED reading :GA#/:GZ# entirely and just slept 2 seconds.
   - telescope_simulator._read_loop() had the same skip pattern.
   - Result: During GoTo, position display froze at the pre-slew position.
   - Fix: Removed the "if not is_slewing:" guard in both bridge and simulator.
     Now ALWAYS reads position regardless of slew state. Added is_slewing
     parameter to the on_altaz_update callback signature (3rd arg).

3. No slew state tracking or UI feedback

Task 2: :D# Slew Detection Broken on Real OnStep via WiFi/TCP

Root cause analysis: During real hardware testing, is_slewing never became
True even during active GoTo. The :D# command response was unreliable over
WiFi/TCP (Discovery #25).

Fix: Triple-method slew detection:
  A. :D# (standard LX200) -- kept as-is, checks for '|' in response
  B. :GU# (OnStep extended status) -- NEW. More reliable than :D#.
     Returns multi-char status string where lowercase 'n' = GoTo active,
     uppercase 'N' = not slewing. Parsed and OR'd with :D# result.
  C. Position delta (firmware-independent) -- NEW. Compares Alt/Az between
     successive polls. If position changes by > 0.3 degrees between polls,
     mount is slewing regardless of :D# or :GU# responses. Handles azimuth
     wrap-around (359->1 = 2 deg, not 358 deg).

All three methods are OR'd together: is_slewing = d_slewing OR gu_slewing
OR position_moving. Position-based detection applied after coordinates are
read (not before, unlike :D# and :GU# which are checked first).

Applied to BOTH serial and TCP transport paths for consistency.

Task 3: Local Plate Solver Accuracy Improvements

Four targeted improvements to the offline triangle/quad solver:

1. Multi-FOV search
   - Previous: Single fixed FOV (default 5 degrees). If wrong, no match.
   - Now: Tries FOV candidates: [fov, fov*0.5, fov*2, fov*0.25, fov*3].
     Handles unknown camera setups where FOV estimate is wrong.
     Phone cameras typically have 50-70 deg FOV, telescopes 0.5-5 deg.

2. Better star detection (hot pixel + elongation rejection)
   - Hot pixel filter: Rejects peaks where < 3 neighboring pixels are above
     threshold (single-pixel cosmic rays or hot pixels)
   - Elongation filter: Computes 2nd-order moments (variance of x/y positions).
     Rejects peaks with elongation > 4x (satellite trails, diffraction spikes)
   - Larger centroiding window: half_win 3 -> 4 for better sub-pixel accuracy

3. Full catalog loading
   - Previous: Loaded only stars.h + messier.h + ngc_select_c.h + caldwell.h
   - Now: Loads ALL .h files in catalogs/data/ directory (25+ files). More
     catalog stars = better chance of matching in sparse fields.
   - Duplicate stars harmless (quad matcher handles them naturally)

4. Adaptive quad matching tolerance
   - Previous: Single tolerance 0.06 for all images
   - Now: Tries tight (0.04) first, relaxes to 0.08 if < 3 matches, then
     to 0.12 as final attempt. Tight tolerance produces fewer false matches
     on clean images; relaxed tolerance works on noisy images.

  40. Server-Sent Events (SSE) via Flask is straightforward: return a
      Response with mimetype="text/event-stream" and a generator that
      yields "data: {...}\n\n" strings.  Works natively in Android WebView

      its duration, but for a single-user app this is fine.

Bugs Found & Fixed
===================

  BUG 1: LinkedHashMap$LinkedKeySet not iterable (local_solver.py)
  ----------------------------------------------------------------
  Symptom:  GET /api/solver/databases returns {"databases":{}} -- empty list.
            Logcat: "Error getting database status: 'LinkedHashMap$LinkedKeySet'
            object is not iterable"
  Cause:    Chaquopy does NOT make Java's LinkedHashMap$LinkedKeySet directly
            iterable in Python for-loops.  The code `for name in java_status.keySet()`
            fails because keySet() returns a Java Set that Python can't iterate.
  Fix:      Use Java Iterator interface explicitly:
              key_iter = java_status.keySet().iterator()
              while key_iter.hasNext():
                  name = key_iter.next()
  File:     android app/app/src/main/python/android_bridge/local_solver.py (line ~191)

  BUG 2: JavaScript innerHTML null reference (web_server.py)
  ----------------------------------------------------------
  Symptom:  WebView console: "Cannot set properties of null (setting 'innerHTML')"
            at page line 2479 (camRefreshAscom function).
  Cause:    On Android, the first pollStatus() call removes the ASCOM camera
            fields from the DOM (cam-ascom-fields.remove() at line 3974).
            But setTimeout(camRefreshAscom, 2000) at line 5196 fires 2 seconds
            later and tries to access lv-ascom-id which no longer exists.
  Fix:      Added null guard: `if (!sel) return;` at the top of camRefreshAscom().
  File:     TelescopeController/web_server.py (line ~4976, synced to Android)

  BUG 3: XZ decompression fails -- no star database files extracted
  -----------------------------------------------------------------
  Symptom:  Database download completes (101 MB) but extraction fails:
            "No star database files extracted from archive".
  Cause:    ASTAP .deb files contain data.tar.xz (XZ/LZMA2 compressed).
            Android's standard library lacks XZ decompression.  The fallback
            to shell `xz` command fails because the device doesn't have it.
  Fix:      Added org.tukaani:xz:1.9 pure Java library (~130 KB, Apache 2.0).
            Updated AstapDatabaseManager.kt to use XZInputStream directly
            instead of shell xz command.  Works on all Android devices.
  Files:    - android app/app/build.gradle.kts (added dependency)
            - android app/app/src/main/java/.../AstapDatabaseManager.kt
              (import XZInputStream, use in extractFromTar + extractByRawScan)

Discoveries
===========
  43. Chaquopy Java-to-Python collection bridging is incomplete: Java Set
      implementations (LinkedHashMap$LinkedKeySet, TreeSet, etc.) are NOT
      directly iterable in Python for-loops.  Always use .iterator() +

      pollStatus() detects platform=android (ASCOM fields, desktop-only UI).

      must have null guards, since the elements may not exist by the time
      the callback fires.

Discoveries
===========
   49. The build system has a syncPythonSources Gradle Sync task (build.gradle.kts
       line 137-156) that copies Python files FROM the parent TelescopeController/
       directory INTO app/src/main/python/ before every build. This means the
       parent directory files are the source of truth for all shared Python modules.
       Editing app/src/main/python/*.py directly is futile -- changes get overwritten
       on the next build. Always edit the parent copies.

   50. The UserGuide HTML is synced from the android app root directory
       (TelescopeController_UserGuide.html) into src/main/python/ by the same
       syncPythonSources task (build.gradle.kts line 147-149).

   51. The activeElement check on <select> elements does not work reliably on
       Android WebView. When the user taps a native dropdown picker, selects a
       value, and the picker closes, the <select> may no longer be the active
       element. The next poll cycle then overwrites the value. The proper fix is
       to either (a) not poll-sync those fields when disconnected, or (b) push
       changes to the backend immediately on user interaction.

    52. A clean build (gradlew clean assembleDebug) is required after editing
        parent Python sources to ensure syncPythonSources copies fresh files.
        An incremental build may use cached assets from the previous build.

Discoveries
===========
   53. Chart.js v4.4.7 UMD bundle is 206KB. Bundling it as a Python-directory
       file served via Flask /static/ route works because Chaquopy packages all
       files in app/src/main/python/ into the APK assets. The Flask route reads
       the file with send_file() at runtime.

   54. Android Camera2 API provides SENSOR_INFO_PHYSICAL_SIZE (SizeF in mm) and
       SENSOR_INFO_ACTIVE_ARRAY_SIZE (Rect in px) via CameraCharacteristics. This
       gives the physical sensor dimensions needed for FOV calculation without
       requiring a lookup table. Pixel size can be derived: pixelSize_um =
       (width_mm / resolution_x) * 1000.

   55. The pier_side and meridian limit code was entirely GEM-specific (German
       Equatorial Mount). For an Alt-Az-only app, this code was dead weight:
       - Alt-Az mounts always report pier_side = 'None' (or 'o' flag in :GU#)
       - Meridian limits (:SXE9/:SXEA) are only meaningful for GEM tracking
         past the meridian (pier flip management)
       - No other code in the app depended on pier_side or meridian values

   56. The MSYS_NO_PATHCONV=1 environment variable is required when running adb
       push/pull with /sdcard/ paths from Git Bash on Windows. Without it, Git
       Bash's POSIX path conversion rewrites /sdcard/Download/ to
       C:/Program Files/Git/sdcard/Download/ which fails.

   57. The HTML tab-content divs do not need to be in any particular order for
       the JS tab-switching to work (it uses display:none/block by ID). But
       grouping them by bottom nav section makes the source code much more
       maintainable.
